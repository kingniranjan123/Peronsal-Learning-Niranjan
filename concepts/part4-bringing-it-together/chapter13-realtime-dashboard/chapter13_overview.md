# Chapter 13 - Case Study: Real-Time Dashboard

**A comprehensive capstone project integrating Kafka, Spark Streaming, WebSockets, and D3.js to build an end-to-end real-time analytics pipeline.**

## Why It Matters

In modern data engineering, building a real-time dashboard is the ultimate test of a streaming architecture. Businesses no longer want to wait hours or days to see how their systems are performing or how users are interacting with their platforms. A real-time dashboard provides immediate visibility into critical metrics—like requests per second, error rates, and concurrent user counts—allowing operations teams to detect anomalies, marketing teams to track campaign launches, and product managers to monitor user engagement as it happens. This case study matters because it takes isolated technologies (message brokers, distributed processing engines, web protocols, and frontend visualization libraries) and weaves them together into a single, cohesive, production-grade application that reflects real-world engineering challenges.

## How It Works

This capstone project is an end-to-end real-time analytics pipeline designed to process web server logs as they are generated and display aggregated metrics on a live dashboard. The architecture is composed of four distinct layers: data generation, message brokering, stream processing, and client presentation. Each layer is decoupled, allowing the system to scale horizontally and handle bursts in traffic without cascading failures. 

The first stage is data generation, where a Log Simulator acts as a stand-in for real web servers. It continuously produces fake, yet realistic, HTTP access logs containing IP addresses, timestamps, URLs, HTTP status codes, and payload sizes. Instead of writing these logs to a flat file, the simulator publishes them directly to an Apache Kafka topic (e.g., `weblogs`). Kafka acts as the highly durable, high-throughput nervous system of our architecture, buffering the incoming streams of logs so that downstream consumers can read them at their own pace.

The second stage is the Spark Streaming Application, the brain of our pipeline. It connects to the `weblogs` Kafka topic using the Direct Stream approach, ensuring exactly-once semantics. Every 5 seconds, Spark pulls the latest batch of logs, parses the raw strings into structured data, and computes real-time metrics such as total requests per second, the top 10 most visited URLs, and the percentage of error responses (HTTP 4xx and 5xx). Once these aggregations are complete, Spark serializes the results into JSON and pushes them to a second Kafka topic (e.g., `stats`), completing the distributed processing loop.

The final stages involve serving this data to the end user. A lightweight WebSocket server (written in Node.js or Java) subscribes to the `stats` Kafka topic. Whenever Spark publishes a new JSON metric payload, the server instantly broadcasts it to all connected browser clients via WebSockets, avoiding the overhead of traditional HTTP polling. On the frontend, a Single Page Application (SPA) uses D3.js to listen for these WebSocket messages and dynamically updates visualizations—such as line charts for traffic volume and bar charts for top URLs—creating a seamless, real-time dashboard experience.

## Data Visualization

The following table illustrates how raw log data transforms at each stage of the pipeline until it becomes a final JSON payload for the dashboard.

| Stage | Data Format / Example | Description |
|---|---|---|
| **1. Generation** | `192.168.1.10 - - [18/Jul/2026:20:22:14 +0000] "GET /home HTTP/1.1" 200 1024` | Raw Apache access log string generated by simulator. |
| **2. Kafka (weblogs)** | `Key: null, Value: "192.168... GET /home..."` | Raw string stored as bytes in a Kafka partition. |
| **3. Spark Parsing** | `LogRecord(ip="192.168.1.10", method="GET", url="/home", status=200, bytes=1024)` | Parsed into structured case class / DataFrame row. |
| **4. Spark Aggregation** | `("/home", 150), ("/api/login", 45), ("/contact", 12)` | Reduced by key to count occurrences per URL. |
| **5. Spark JSON Output** | `{"timestamp": 1690000000, "top_urls": [{"url": "/home", "count": 150}]}` | Final aggregated batch serialized to JSON. |
| **6. Kafka (stats)** | `Key: null, Value: {"timestamp": ...}` | JSON string buffered for the WebSocket server. |
| **7. WebSocket Output** | `{"type": "METRICS_UPDATE", "data": {...}}` | JSON pushed over TCP socket to the browser. |
| **8. D3.js Render** | Bar chart height changes from `100px` to `150px` | Data bound to DOM element resulting in visual update. |

## Code Example

Below is a high-level representation of how the components piece together. For full code, refer to the individual topics in this chapter.

```scala
// --- High Level Pipeline Overview ---

// 1. Setup Streaming Context
val sparkConf = new SparkConf().setAppName("RealTimeDashboard")
val ssc = new StreamingContext(sparkConf, Seconds(5))

// 2. Consume from Kafka (Log Simulator -> weblogs)
val kafkaParams = Map[String, Object](
  "bootstrap.servers" -> "localhost:9092",
  "key.deserializer" -> classOf[StringDeserializer],
  "value.deserializer" -> classOf[StringDeserializer],
  "group.id" -> "dashboard-group",
  "auto.offset.reset" -> "latest"
)
val topics = Array("weblogs")
val stream = KafkaUtils.createDirectStream[String, String](
  ssc,
  LocationStrategies.PreferConsistent,
  ConsumerStrategies.Subscribe[String, String](topics, kafkaParams)
)

// 3. Process the stream
val logLines = stream.map(record => record.value)
val parsedLogs = logLines.map(LogParser.parse) // Custom parser

val urlCounts = parsedLogs
  .map(log => (log.url, 1))
  .reduceByKey(_ + _)
  .map { case (url, count) => s"""{"url": "$url", "count": $count}""" }

// 4. Publish to Kafka (stats -> WebSocket Server)
urlCounts.foreachRDD { rdd =>
  rdd.foreachPartition { partitionOfRecords =>
    // Create Kafka producer inside partition to avoid serialization issues
    val producer = new KafkaProducer[String, String](ProducerConfig.get())
    partitionOfRecords.foreach { json =>
      val message = new ProducerRecord[String, String]("stats", null, json)
      producer.send(message)
    }
    producer.close()
  }
}

ssc.start()
ssc.awaitTermination()
```

## Common Pitfalls

* **Producer Initialization:** Attempting to create the `KafkaProducer` on the Spark driver and passing it to worker nodes will throw `NotSerializableException`. The producer must be created inside `foreachPartition`.
* **Micro-batch Bottlenecks:** If the processing time for a 5-second batch consistently exceeds 5 seconds, the streaming application will experience scheduling delays and eventually crash with an OutOfMemoryError.
* **WebSocket Connection Leaks:** Failing to properly handle client disconnects on the WebSocket server can lead to memory leaks and socket exhaustion, bringing down the presentation layer.
* **Data Skew:** If one URL (e.g., `/`) receives 99% of the traffic, the Spark partition handling that key will become a bottleneck. Salting keys might be required for extreme skew.
* **Browser DOM Overload:** D3.js binding thousands of data points every second will freeze the browser. Data must be aggregated on the backend (Spark) before sending to the frontend.

## Key Takeaway

A real-time dashboard is an orchestration of decoupled systems where message brokers provide the buffer, stream processing provides the intelligence, and WebSockets provide the low-latency delivery mechanism for dynamic visualization.
